# Repo Index — Renaissance AI and Education Resource Hub

> Map of the public repo. If you're new here (human or cold-starting Claude session), start with this file.

The hub is a **referatory** — a curated index of evidence-based K-12, higher-education, and learning-engineering resources, hosted at GitHub Pages and consumable by both humans and LLM agents (via `llms.txt` and an MCP server). It does not store source content; it stores metadata + descriptions and links out.

**Current state:** 2,445 indexed entries across 20+ sources (WWC, Mathematica, LPI, Digital Promise, Evidence for ESSA, NWEA, Campbell, AIMS, and more). Coverage tracked in `docs/data.json` (`meta.coverage`) and `data/source-targets.json`.

## File map

### Root
- **`README.md`** — human-facing repo description.
- **`index.md`** — this file.
- **`.gitignore`** — note: `wiki/`, `docs/legacy/`, `docs/staging/` are gitignored.

### `docs/` — the published referatory (GitHub Pages root)

Only published output lives here — no scripts, no build tooling.

- **`llms-full.txt`** — primary file. All entries with descriptions + auto-generated nav header. The canonical corpus.
- **`llms.txt`** — compact index (titles, URLs, types, tags — no descriptions).
- **`data.json`** — JSON view; consumed by `index.html` and the MCP worker.
- **`index.html`** — human-facing UI (browse by tag, source, type).
- **`tags/`** — per-tag markdown files (auto-generated by `build_from_db.py`).
- **`schema.md`** — entry format, type taxonomy, full tag vocabulary.
- **`purpose.md`** — scope, audience, hosting notes.
- **`gem-knowledge.txt`** — generated artifact for the Gemini Gem.
- **`staging/`** — gitignored temp working directory for scraped data before processing.

### `scripts/` — Python tooling

All executable scripts. Run from repo root: `python scripts/{script}.py`.

- **`build_from_db.py`** — builds all published outputs (llms-full.txt, llms.txt, data.json, tags/, gem-knowledge.txt) from `data/hub.db`. **Run after every data change.**
- **`scrape.py`** — config-driven scraper. Reads source configs from `sources/`, outputs to `docs/staging/`.
- **`process_staged.py`** — processes staged JSON into `data/hub.db`. Handles tagging and DB insertion.
- **`verify_urls.py`** — domain-aware URL checker with throttling. Writes results to `data/hub.db` and `data/broken-urls.json`.
- **`source_check.py`** — pre-flight accessibility probe for all sources.
- **`playwright_scrape.py`** — scraper for JS-rendered sources (TNTP, Digital Promise).
- **`list_sources.py`** — utility for listing sources from hub.db.

### `sources/` — per-source scraping configs and profiles

Each source has a `.json` config (consumed by `scrape.py`) and a `.md` profile (human-readable documentation of discovery method, access, scope, quirks). Some have `-backlog.txt` files tracking items that couldn't be fully indexed.

- **`{source}.json`** — scraping config (discovery URL, selectors, API params, pagination).
- **`{source}.md`** — source profile (discovery method, access notes, scope, scraping instructions).
- **`{source}-backlog.txt`** — URLs/items that need manual review or are missing descriptions.

### `data/` — database and data files

- **`hub.db`** — SQLite database. Single source of truth for all entries. All published files are generated from this.
- **`source-targets.json`** — coverage targets per source (known totals, priority, status).
- **`broken-urls.json`** — known broken/flagged URLs from `verify_urls.py`.
- **`url-verification.json`** — URL verification results.

### `meta/` — operational docs and guides

No scripts or data files — just documentation for operators and agents.

- **`agent-guide.md`** — master operational reference for cold-starting Claude agents.
- **`operator-guide.md`** — deployment surfaces, manual deploy steps, after-merge checklist.
- **`automation-prompt.md`** — prompt for the weekly automated source-check routine.
- **`automation-log.md`** — append-only log of weekly automation runs.
- **`backlog-prompt.md`** — prompt for expanding coverage of a single source.
- **`new-source-prompt.md`** — prompt for onboarding a brand-new source.
- **`inclusion-criteria.md`** — what qualifies for inclusion (source-level + resource-level rules).
- **`sources-inventory.md`** — full catalog of sources with access notes and URL patterns.
- **`source-audit.md`** — accessibility review of every source.
- **`sources-log.md`** — raw log of source-access attempts.
- **`processing-log.md`** — auto-appended by `process_staged.py`.
- **`gem-instructions.md`** — instructions for the Google Gemini Gem.
- **`playwright-guide.md`** — Playwright installation and usage reference.

### `worker/` — Cloudflare Worker (MCP server)
- Deployed at `https://renaissance-hub.joon-96a.workers.dev`. Exposes the hub via MCP so LLM agents can query it directly.
- Imports `docs/data.json` at build time — redeploy after data updates: `cd worker && npx wrangler deploy`.

## Workflow

### Adding entries from an existing source

```
1. Check source profile          sources/{source}.md — read access notes, quirks, robots.txt
2. Scrape                        python scripts/scrape.py {source}
                                 → outputs docs/staging/{source}.json
                                 → backlog items go to sources/{source}-backlog.txt
3. Process into database         python scripts/process_staged.py {source}
                                 → inserts into data/hub.db with auto-tagging
                                 → logs to meta/processing-log.md
4. Rebuild published files       python scripts/build_from_db.py
                                 → regenerates docs/llms-full.txt, llms.txt, data.json,
                                   tags/, gem-knowledge.txt
5. Verify (optional)             python -m http.server 8765   (from docs/)
                                 → check index.html, spot-check entries
6. Commit + deploy               git add + commit
                                 → GitHub Pages auto-deploys docs/
                                 → MCP worker needs manual redeploy:
                                   cd worker && npx wrangler deploy
                                 → Gemini Gem needs manual upload of
                                   docs/gem-knowledge.txt
```

### Onboarding a new source

1. Run the interactive prompt: `meta/new-source-prompt.md`
2. Creates: `sources/{source}.json` (config), `sources/{source}.md` (profile)
3. Updates: `data/source-targets.json`, `meta/sources-inventory.md`, `meta/source-audit.md`
4. Then follow the "Adding entries" workflow above.

### Weekly automated check

Runs via `meta/automation-prompt.md` (cloud routine or interactive). Checks all sources for new content, stages new entries, builds, commits, and opens a PR. Results logged to `meta/automation-log.md`.

### URL verification

```
python scripts/verify_urls.py                    # all unverified
python scripts/verify_urls.py --source {name}    # one source
python scripts/verify_urls.py --recheck          # re-check broken/flagged
python scripts/verify_urls.py --auto-exclude     # auto-exclude confirmed 404/410s
```

Domain-aware throttling (5s between same-domain requests). Results written to `data/hub.db` and `data/broken-urls.json`. After excluding broken URLs, rebuild with `python scripts/build_from_db.py`.

| Surface | Auto-updates? | Manual step |
|---|---|---|

| **MCP server** (Cloudflare Worker) | No | `cd worker && npx wrangler deploy` |
| **Gemini Gem** | No | Upload `docs/gem-knowledge.txt` in Gemini console |

## Common operations

```powershell
# Rebuild all published files from hub.db
python scripts/build_from_db.py

# Scrape a source
python scripts/scrape.py {source}              # full scrape
python scripts/scrape.py {source} --test       # test selectors
python scripts/scrape.py {source} --diff       # show only new items

# Process staged data into hub.db
python scripts/process_staged.py {source}

# Verify URLs
python scripts/verify_urls.py --source {name}  # one source
python scripts/verify_urls.py --all            # everything

# Local human-facing UI
cd docs && python -m http.server 8765

# Deploy the MCP worker
cd worker && npx wrangler deploy
```

## Conventions

- **No inference.** Descriptions must come from fetched page content, never the title alone. Drop entries that 404 or 403.
- **UTF-8 only.** All files. `build_from_db.py` validates and rejects mojibake.
- **Subagents write to files, not text.** Any agent collecting entries writes to `docs/staging/{source}.txt`. Data returned as agent text is lost on context compaction.
- **Scrape → process → build → commit.** `python scripts/scrape.py {source}` → `python scripts/process_staged.py {source}` → `python scripts/build_from_db.py` → commit.
- **Structure changes update this index.** Any file/folder move, rename, addition, or deletion updates this file.
